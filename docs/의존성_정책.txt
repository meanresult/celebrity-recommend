insta_pipeline 의존성 관리 기준

이 프로젝트는 Airflow, dbt, Streamlit 컨테이너를 각각 따로 운영하므로,
런타임 의존성과 로컬 개발 의존성을 분리해서 관리한다.

==================================================
1. 기본 원칙
==================================================

1) 컨테이너 런타임 의존성
- `requirements.txt`: Airflow + extractor 런타임 기준
- `requirements_dbt.txt`: dbt 컨테이너 런타임 기준
- `requirements_streamlit.txt`: Streamlit 컨테이너 런타임 기준

2) 로컬 개발환경 의존성
- `pyproject.toml`: 로컬 개발 메타데이터와 개발용 Python 버전 기준
- `uv.lock`: 로컬 개발환경 재현용 잠금 파일

3) 운영 기준
- 컨테이너 내부에 설치되는 패키지는 반드시 각 `requirements*.txt`를 기준으로 본다.
- `pyproject.toml`은 컨테이너 배포 기준 파일이 아니다.

==================================================
2. 왜 이렇게 나누는가
==================================================

[이유 1. 컨테이너 역할이 다르다]
- Airflow 컨테이너에는 Airflow provider, Playwright, Snowflake connector가 필요하다.
- dbt 컨테이너에는 dbt-core, dbt-snowflake만 있으면 된다.
- Streamlit 컨테이너에는 streamlit, plotly, pandas 같은 UI/조회용 패키지가 필요하다.

즉, 세 컨테이너는 설치해야 하는 패키지가 서로 다르다.
하나의 공통 런타임 파일로 억지로 묶으면 이미지가 비대해지고 충돌 가능성도 커진다.

[이유 2. 로컬 개발 도구와 런타임 의존성은 목적이 다르다]
- 로컬 개발에서는 버전 고정, 도구 통일, 환경 재현이 중요하다.
- 컨테이너 런타임에서는 실제 서비스 실행에 필요한 패키지만 안정적으로 설치되는 것이 더 중요하다.

==================================================
3. 이 기준을 안 지키면 생기는 문제
==================================================

- 어떤 파일이 진짜 기준인지 헷갈린다.
- Docker에서는 되는데 로컬 uv 환경에서는 안 되거나, 그 반대 상황이 생긴다.
- 팀원이 의존성을 추가할 때 어디를 수정해야 하는지 몰라 중복 수정이 발생한다.
- pyproject와 requirements의 버전이 어긋나면 장애 원인 추적이 어려워진다.
- 나중에 CI를 붙일 때도 설치 기준이 두 군데로 갈라져 파이프라인이 복잡해진다.

==================================================
4. 앞으로의 수정 규칙
==================================================

[런타임 패키지를 추가하거나 수정할 때]
- 먼저 해당 컨테이너의 `requirements*.txt`를 수정한다.
- 예:
  - Airflow/크롤러에 필요한 패키지면 `requirements.txt`
  - dbt 관련이면 `requirements_dbt.txt`
  - Streamlit 관련이면 `requirements_streamlit.txt`

[로컬 개발환경만 바꿀 때]
- `pyproject.toml`을 수정한다.
- 필요하면 `uv.lock`도 함께 갱신한다.

[Python 버전을 바꿀 때]
- `pyproject.toml`
- `.python-version`
- Dockerfile 또는 컨테이너 베이스 이미지
- CI 설정
위 네 곳이 서로 일치하는지 확인한다.

==================================================
5. 현재 프로젝트에 적용한 기준
==================================================

- `pyproject.toml`은 개발 메타데이터 전용으로 축소한다.
- Python 버전은 호환성을 고려해 `3.11` 기준으로 맞춘다.
- 컨테이너 런타임 의존성은 기존처럼 `requirements*.txt`로 유지한다.
- README에는 이 역할 분리를 명확히 적는다.

==================================================
6. 한 줄 요약
==================================================

- 컨테이너에서 돌아가는 패키지는 `requirements*.txt`가 기준이다.
- 로컬 개발환경 재현은 `pyproject.toml` + `uv.lock`이 맡는다.
- 둘 다 유지할 수는 있지만, 역할이 겹치지 않게 관리해야 한다.
