# Instagram Popup Load Failure Review

이 문서는 현재 `legacy/extractors/main_mini_v10.py`로 이동된 extractor에서 발생한 `팝업 로드 실패` 이슈를 정리한 회고 문서다.

문제의 직접 증상은 아래와 같았다.

- tagged 페이지 진입은 성공한다.
- 게시물 URL 스냅샷도 정상적으로 수집된다.
- 그러나 각 게시물을 클릭한 뒤 `article` 셀렉터가 뜨지 않아 모두 `⚠️ 팝업 로드 실패`로 처리된다.
- 결과적으로 수집 건수가 `0`건으로 끝난다.

## 1. 문제였던 부분

문제가 발생한 코드는 아래 구간이다.

- 파일: `legacy/extractors/main_mini_v10.py`
- 동작:
  1. tagged 페이지에서 게시물 링크 수집
  2. JavaScript로 링크 클릭
  3. `page.wait_for_selector("article", state="visible", timeout=5000)` 대기
  4. 실패 시 팝업 로드 실패 처리

여기서 핵심 문제는 "팝업이 안 열린 것인지", "팝업은 열렸지만 우리가 기다리는 DOM이 바뀐 것인지", "로그인/차단 페이지로 튄 것인지"를 로그만으로 구분할 수 없었다는 점이다.

즉, 실패는 보였지만 원인은 보이지 않는 상태였다.

## 2. 확인한 사실

이번 점검에서 확인한 내용은 아래와 같다.

1. Airflow 로그상 여러 게시물 URL에서 동일한 실패가 반복되었다.
2. 특정 게시물만 문제가 아니라 거의 전부 실패했다.
3. 브라우저에서 `https://www.instagram.com/amomento.co/tagged/`를 비로그인 상태로 열면 Instagram 로그인 페이지로 리다이렉트된다.
4. 따라서 세션 만료, 클릭 방식 문제, DOM 구조 변경, 자동화 제한 가능성을 모두 열어둬야 했다.

## 3. 생각해본 해결 방향

이번 이슈에 대해 바로 떠올릴 수 있는 해결 방향은 크게 3가지였다.

### 방향 1. 셀렉터만 바로 바꾸기

예를 들면 `article` 대신 다른 상세 페이지 셀렉터를 기다리도록 수정하는 방법이다.

장점:
- 수정량이 작다.
- 운 좋으면 바로 복구될 수 있다.

단점:
- 실제 원인이 클릭 실패나 로그인 리다이렉트면 효과가 없다.
- 근거 없이 셀렉터만 바꾸면 또 다른 형태의 실패를 만들 수 있다.

### 방향 2. 클릭 방식을 바꾸기

현재는 `page.evaluate(... link.click())` 방식으로 클릭하고 있다.  
이걸 Playwright locator click, keyboard navigation, URL 직접 이동 등으로 바꿔볼 수 있다.

장점:
- Instagram SPA 라우팅 문제를 해결할 가능성이 있다.

단점:
- 이것도 원인 확정 전에는 추측성 수정이다.
- 클릭 방식이 문제인지 DOM이 문제인지 분리되지 않은 상태에서 바꾸면 디버깅이 더 어려워질 수 있다.

### 방향 3. 실패 증적을 먼저 남기기

팝업 실패 시 아래 정보를 남기는 방법이다.

- 현재 URL
- 실제 예외 메시지
- 실패 시점 스크린샷

장점:
- 로그인 페이지로 튀는지, 차단 페이지가 뜨는지, 그냥 grid 화면에 머무는지 바로 구분 가능하다.
- 다음 수정이 추측이 아니라 근거 기반이 된다.

단점:
- 즉시 복구는 아니다.
- 한 번 더 실행해서 증적을 모아야 한다.

## 4. 가장 합리적이라고 본 선택

가장 합리적인 선택은 `방향 3. 실패 증적을 먼저 남기기`였다.

이유는 단순하다.

1. 현재 실패는 "팝업이 안 떴다"는 현상만 보이고, 원인은 여러 개로 열려 있었다.
2. 이 상태에서 셀렉터나 클릭 방식을 먼저 바꾸면, 문제를 푸는 게 아니라 증상을 덮을 가능성이 있었다.
3. 운영 크롤러는 한 번 성공하는 코드보다, 실패 원인을 빠르게 식별할 수 있는 코드가 더 중요하다.

즉, 이번 단계에서는 복구보다 `원인 분리`가 우선이었다.

## 5. 실제로 선택한 조치

당시 `extractors/main_mini_v10.py`에 추가했던 디버그 로직은 현재 `legacy/extractors/main_mini_v10.py`에 보관되어 있다.

추가한 내용:

1. 팝업 실패 시 현재 `page.url` 출력
2. 팝업 실패 시 예외 메시지 출력
3. 팝업 실패 시 스크린샷 저장
4. 스크린샷 저장 경로를 로그에 출력

저장 경로:

- `/tmp/insta_debug`

이 조치로 다음 실행부터는 아래 구분이 가능해진다.

- 로그인 페이지로 리다이렉트되는지
- 차단 또는 경고 페이지가 뜨는지
- 상세 팝업 대신 다른 레이아웃으로 열리는지
- 클릭 자체가 무시되고 원래 grid에 머무는지

## 6. 이번 선택이 실무적으로 의미 있는 이유

운영 이슈에서 가장 위험한 패턴은 "원인도 모른 채 추측으로 계속 코드를 바꾸는 것"이다.

이번 조치는 문제를 바로 고치는 수정이라기보다, 다음 수정이 정확해지도록 만드는 관측 장치 추가에 가깝다.

실무에서는 이런 순서가 더 안전하다.

1. 실패 지점 식별
2. 증적 수집
3. 원인 분리
4. 가장 작은 수정으로 복구
5. 재발 방지 로깅 유지

## 7. 다음 단계

다음 실행 후 아래 로그를 기준으로 원인을 확정한다.

1. `현재 URL`
2. `팝업 로드 에러`
3. `디버그 스크린샷`

그 결과에 따라 후속 조치는 달라진다.

- 로그인 페이지로 튀면:
  - `storage_state.json` 세션 갱신
  - 로그인 상태 검증 강화
- grid 화면에 그대로 머무르면:
  - 클릭 방식을 `locator.click()` 또는 직접 URL 이동 방식으로 재검토
- 상세는 열렸는데 `article`만 없으면:
  - 상세 페이지 판별 셀렉터를 재설계
- 경고/차단 화면이면:
  - headless 설정, 요청 속도, 세션 전략 재검토

## 8. 정리

이번 이슈의 본질은 단순한 selector failure가 아니라, "실패 원인을 구분할 증적이 없었다"는 점이었다.

그래서 이번 수정의 목적은 문제를 감으로 고치는 것이 아니라, 다음 실행에서 정확한 원인을 드러내는 것이었다.

이 선택이 가장 합리적이었던 이유는, 가장 적은 변경으로 가장 큰 정보를 추가할 수 있었기 때문이다.
