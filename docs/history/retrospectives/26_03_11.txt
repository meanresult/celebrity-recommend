2026-03-11 작업 회고

1. 오늘 무엇이 문제였나

- Airflow와 dbt, Streamlit이 같은 Snowflake를 보더라도 각각 보는 위치와 설정 방식이 달랐다.
- raw 데이터는 쌓이는데, dbt 결과가 어디에 저장되는지와 Streamlit이 어디를 읽는지가 명확하지 않았다.
- dbt 모델 이름과 실제 계산 의미가 일부 맞지 않았고, stage 모델에 필요한 맥락 컬럼도 부족했다.
- 키를 문자열 이어붙이기로 만들고 있어, 규칙이 조금만 바뀌어도 나중에 혼란이 생길 수 있었다.
- 설치 문서가 점점 복잡해지는데, 처음 보는 사람 입장에서는 “내가 뭘 직접 해야 하는지”가 바로 보이지 않았다.

2. 그대로 두면 생기는 문제

- 대시보드 숫자는 나오는데, 그 숫자를 잘못 이해할 수 있다.
- dbt 결과가 STAGE와 MART에 섞여 쌓이면 운영자도 어떤 테이블이 중간 결과인지 최종 결과인지 헷갈리게 된다.
- 나중에 브랜드별, 날짜별 분석을 더 하고 싶을 때 raw 테이블을 다시 뒤져야 해서 작업이 반복된다.
- 키 생성 규칙이 흔들리면 같은 데이터도 다른 데이터처럼 보일 수 있다.
- 문서가 불명확하면 새로 들어온 사람은 환경설정에서 시간을 많이 쓰고, 잘못된 계정이나 스키마를 붙잡고 헤매게 된다.

3. 해결하기 위해 생각한 방안

- 방안 A: 지금 구조를 유지하고 문서만 더 보강한다.
- 방안 B: 저장 구조와 의미를 실제로 분리한다. 즉, RAW_DATA / STAGE / MART 역할을 코드와 문서에 모두 반영한다.

선택:
- 방안 B 선택

선택 이유:
- 문서만 좋아져도 잠깐은 덜 헷갈릴 수 있다.
- 하지만 실제 저장 구조가 섞여 있으면 시간이 갈수록 다시 혼란이 생긴다.
- 그래서 이번에는 “설명”만 바꾸는 것이 아니라 “실제 구조”를 같이 바꾸는 쪽이 더 맞다고 판단했다.

4. 오늘 실제로 반영한 것

- Airflow raw 적재는 계속 `FSH.RAW_DATA`를 쓰도록 유지했다.
- dbt는 stage 모델을 `FSH.STAGE`, mart 모델을 `FSH.MART`에 저장하도록 분리했다.
- Streamlit은 최종 분석용 mart인 `FSH.MART.cross_brand_accounts`를 읽도록 바꿨다.
- `brand_count`를 `tagged_account_count`로 바꿔서 이름과 실제 의미를 맞췄다.
- `group_by_tagged_post`에 `brand_name`, `brand_id`, `post_date`를 남겨서 나중에 분석 확장이 가능하게 했다.
- `pk_tagged_post`는 문자열 이어붙이기 대신 surrogate key 방식으로 바꿨다.
- dbt 테스트를 추가해 raw와 모델 품질을 확인할 수 있게 했다.
- 설치 문서와 다중 날짜 실행 문서를 정리했고, Snowflake에서 무엇을 직접 실행해야 하는지도 명확히 적었다.

5. 이번에 얻은 교훈

- “설명상 분리”와 “실제 분리”는 다르다.
  폴더만 나눠놓는다고 운영 구조가 분리되는 것은 아니다.

- “돌아가는 코드”와 “오래 유지되는 코드”는 다르다.
  지금 당장 실행되는 것보다, 나중에 다른 사람이 이해할 수 있는 구조가 더 중요해진다.

- 이름은 단순한 장식이 아니다.
  계산식과 컬럼 이름이 어긋나면 숫자가 맞아도 해석은 틀릴 수 있다.

- 테스트는 나중에 붙이는 옵션이 아니라, 데이터가 조용히 망가지는 걸 막는 안전장치다.

- 문서는 개발자를 위한 참고서가 아니라, 처음 보는 사람이 실수하지 않게 하는 안내판이어야 한다.

6. 신입이 특히 생각해봐야 할 점

- 이 컬럼 이름은 정말 계산 의미와 맞는가?
- 이 데이터는 raw인지, 중간 가공인지, 최종 분석용인지 분명한가?
- 지금 편하게 만든 규칙이 나중에도 설명 가능한 규칙인가?
- 새로 들어온 사람이 이 문서만 보고도 같은 환경을 만들 수 있는가?

한 줄 요약:

오늘의 핵심은 “헷갈릴 수 있는 구조를 실제로 분리하고, 그 구조를 사람도 이해할 수 있게 맞춘 것”이다.
