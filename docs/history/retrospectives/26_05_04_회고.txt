1. 문제점 파악

이번 작업에서 가장 먼저 드러난 문제는 Airflow에 예전 실패 작업과 대기 상태 작업이 많이 남아 있었다는 점이다. Airflow는 데이터를 자동으로 수집하고 옮기는 순서를 관리하는 도구인데, 이 안에 실패 기록과 멈춘 기록이 쌓이면 "지금 무엇이 새로 돌아가는 중인지"와 "예전에 꼬여서 남아 있는 것인지"를 한눈에 구분하기 어려워진다. 실제로 오늘 확인했을 때는 실패로 끝난 실행 기록이 많이 누적되어 있었고, 일부 작업은 끝나지 않은 것처럼 `running`이나 `scheduled` 상태로 남아 있었다.

또 하나의 문제는 날짜 해석이 직관적이지 않았다는 점이다. 사용자는 "오늘 데이터를 다시 넣고 싶다"고 생각하지만, Airflow 내부에서는 UTC 기준 실행 시간과 한국 시간 기준 처리 날짜가 다르게 보일 수 있다. 이 차이를 바로 이해하지 못하면 오늘 작업을 다시 돌린다고 했는데 실제로는 어제 기준 실행 기록을 만지게 되는 상황이 생길 수 있다.

2. 문제점이 일으킬 상황

이런 상태가 계속되면 운영자가 대시보드를 봤을 때 현재 파이프라인이 정말로 돌고 있는지, 아니면 예전 작업이 남아서 그렇게 보이는지 판단하기 어려워진다. 그러면 문제를 고치기보다 화면을 해석하는 데 더 많은 시간을 쓰게 된다. 자동화 도구는 사람 일을 줄여주려고 쓰는 것인데, 오히려 사람이 계속 상태를 추적해야 하는 상황이 되어버리는 것이다.

데이터 측면에서도 혼선이 생길 수 있다. 이번 프로젝트는 Instagram 데이터를 수집해서 DuckDB에 넣는 흐름인데, 어떤 날짜 데이터를 다시 넣는지 기준이 흔들리면 "오늘 데이터를 다시 적재했다"는 말 자체가 불명확해진다. 심하면 같은 브랜드를 여러 번 다시 돌리거나, 반대로 필요한 날짜를 놓치게 될 가능성도 있다.

3. 개선 필요성

이번 일은 단순히 버튼 몇 번 잘못 눌러서 생긴 문제가 아니라, 운영 기준이 명확하지 않으면 누구나 반복해서 겪을 수 있는 문제라고 본다. 실패한 작업을 지울 때도 "무조건 다 삭제"가 아니라 "무엇을 지우고, 무엇은 다시 살려야 하는지" 기준이 있어야 한다. 그래야 문제가 생겼을 때도 침착하게 정리하고 다시 시작할 수 있다.

특히 이 프로젝트는 Instagram 수집, Airflow 스케줄링, DuckDB 적재가 이어지는 구조라서 한 단계만 꼬여도 뒤 단계가 연쇄적으로 영향을 받는다. 그래서 작업을 다시 돌릴 때는 단순 재시작보다 상태 정리와 날짜 확인이 먼저다. 운영 과정에서 이 순서를 습관으로 만들 필요가 있다.

4. 개선 방안 리스트

- 실패한 실행 기록과 대기 중인 실행 기록을 구분해서 관리한다. 실패 기록은 정리 대상으로 보고, 아직 살아 있는 작업은 정말 계속 돌릴 것인지 먼저 판단한 뒤 clear 한다.
- "오늘"이라는 표현 대신 항상 한국 시간 날짜와 Airflow 실행 시각을 함께 적는다. 예를 들어 "2026-05-04 KST 데이터 재적재"처럼 적어 혼동을 줄인다.
- 수동 재실행 절차를 문서로 남긴다. 어떤 순서로 상태를 확인하고, 어떤 명령으로 clear 하고, 어떤 시점에 manual trigger를 넣는지 정리하면 다음 작업자가 훨씬 빠르게 대응할 수 있다.
- 재실행 전에 현재 메타데이터 상태를 숫자로 확인하는 습관을 만든다. 실패 건수, 실행 중 건수, 대기 건수를 먼저 보면 감으로 판단하는 일을 줄일 수 있다.

5. 개선 방안 선택과 그 이유

여러 방법 중에서 가장 먼저 적용할 것은 "상태 확인 -> 실패 기록 정리 -> 대기 작업 clear -> 날짜를 명시한 수동 실행" 순서를 운영 기준으로 만드는 것이다. 이 방법은 새로운 시스템을 도입할 필요가 없고, 지금 있는 Airflow와 문서만으로 바로 적용할 수 있다. 비용이 거의 들지 않으면서도 운영 혼선을 크게 줄일 수 있다는 점이 장점이다.

반면, 더 복잡한 모니터링 도구를 새로 붙이거나 구조를 크게 바꾸는 방법은 당장은 부담이 크다. 지금 단계에서는 문제의 핵심이 "도구 부족"보다는 "운영 순서와 판단 기준의 부재"에 가깝다고 본다. 그래서 먼저 절차를 단순하고 명확하게 만드는 쪽이 실용적이고 유지보수에도 유리하다.

6. 성과 지표 도출

- 실패 상태로 남아 있는 Airflow 실행 기록 수가 얼마나 줄었는지 본다.
- 수동 재실행 시 "오늘 기준 데이터"를 잘못 선택하는 횟수가 0건으로 유지되는지 본다.
- 재실행 요청부터 실제 manual trigger 완료까지 걸리는 시간을 측정한다.
- 재실행 후 각 브랜드 DAG의 완료 비율이 얼마나 안정적으로 올라가는지 본다.
- 운영자가 로그를 다시 해석하느라 소비하는 시간이 이전보다 줄었는지 체감과 기록으로 함께 확인한다.

7. 교훈

이번 일을 통해 느낀 점은, 자동화 시스템에서 가장 무서운 것은 "실패" 그 자체보다 "상태를 정확히 읽지 못하는 상황"이라는 것이다. 실패는 고치면 되지만, 상태 해석이 흔들리면 같은 문제를 계속 다른 모습으로 반복하게 된다. 앞으로는 문제가 생기면 바로 다시 돌리기보다 먼저 현재 상태를 숫자와 날짜로 확인하고, 그다음에 정리와 재실행을 순서대로 진행해야겠다.

또한 기술 용어를 아는 사람끼리만 이해할 수 있는 방식으로 기록하면 나중에 다시 봤을 때도 실제 도움이 적을 수 있다. 다음부터는 나 자신이 며칠 뒤 다시 읽어도 바로 이해할 수 있도록, 오늘 어떤 문제가 있었고 왜 이런 순서로 대응했는지를 더 쉬운 말로 남기는 습관을 가져야겠다.
