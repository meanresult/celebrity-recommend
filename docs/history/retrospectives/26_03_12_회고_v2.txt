[문제 1. 화면 문제처럼 보였지만 실제로는 데이터 집계 기준이 어긋난 문제]

문제 포착

처음에는 브랜드 스냅샷 차트와 계정 랭킹이 기대와 다르게 보여서, 화면이 이상하거나 시각화가 잘못된 것처럼 느껴졌다. 특히 "왜 이 브랜드가 너무 적게 보이지?" 또는 "왜 이 계정이 상위에 있지?" 같은 의문이 들었다.

문제 상황 파악

조금 더 들여다보니, 문제는 화면이 아니라 집계 기준에 있었다. 브랜드 스냅샷은 "계정당 1회로 셀 것인가"와 "게시물에서 태그된 횟수까지 더할 것인가"에 따라 결과가 달라졌다. 계정 랭킹도 "전체 태그 수"로 볼지, "선택 브랜드를 실제로 몇 번 태그했는가"로 볼지에 따라 순위가 달라졌다.

이 상태를 그대로 두면 화면은 그럴듯해 보여도 사용자 질문과 전혀 다른 숫자를 보여줄 가능성이 컸다. 예를 들어 사용자는 "이 계정이 amomento를 몇 번 태그했나?"를 보고 싶은데, 시스템은 "이 계정이 전체적으로 몇 개 브랜드를 태그했나?"를 보여주고 있을 수 있었다. 결국 화면 문제가 아니라, 숫자의 뜻이 틀어진 상태였던 셈이다.

해결 방법 도출

- 화면만 먼저 손보는 방법
- 기존 쿼리는 유지하고 설명 문구만 바꾸는 방법
- 집계 기준 자체를 다시 정의하고 쿼리를 재설계하는 방법

해결 방법 선정 및 타당한 이유

세 번째 방법을 선택했다. 이유는 이 문제의 핵심이 "어떻게 보이느냐"가 아니라 "무엇을 세고 있느냐"였기 때문이다. 화면만 바꾸는 방법은 가장 빠르지만, 잘못된 숫자를 더 예쁘게 보여주는 결과가 될 수 있었다. 설명만 바꾸는 방법도 비용은 적게 들지만, 사용자 요구와 다른 집계 기준 자체는 그대로 남는다.

반면 쿼리를 재설계하는 방법은 손이 더 많이 가지만, 한 번 정의를 맞춰두면 이후 차트, 표, KPI, 인사이트 문구까지 같은 기준으로 정리할 수 있다. 유지보수성과 해석의 일관성을 생각했을 때 이쪽이 장기적으로 더 싼 선택이라고 판단했다.

예시 코드

SELECT
    tagged_account,
    COUNT(DISTINCT insta_id) AS account_count,
    SUM(tagged_post_count) AS tagged_post_total
FROM fsh.mart.account_tagged_accounts
GROUP BY tagged_account;

이 코드에서 중요한 부분은 `COUNT(DISTINCT insta_id)`이다. 같은 계정이 같은 브랜드를 여러 번 태그하더라도, "그 브랜드를 태그한 계정 수"는 한 번만 세겠다는 뜻이다.

성과 측정

- 변경 전:
  - 브랜드 스냅샷과 계정 랭킹 기준이 사용자 질문과 정확히 맞지 않았음
  - 표와 차트가 서로 다른 의미로 읽힐 여지가 있었음
- 변경 후:
  - 브랜드 스냅샷은 `계정당 1회` 기준
  - 계정 랭킹은 `선택 브랜드 총 태그 수` 기준
  - 첫 탭 문구와 인사이트도 같은 집계 기준으로 정리

검증 방식과 남은 리스크

검증은 실제 쿼리를 컨테이너 안에서 실행해 `COUNT(DISTINCT insta_id)` 결과와 `SUM(tagged_post_count)` 결과를 함께 확인하는 방식으로 진행했다. 또, Streamlit 화면 문구가 새 집계 기준과 맞는지 직접 확인했다.

다만 남은 리스크도 있다. 실제 데이터 특성상 여러 브랜드의 `account_count`가 대부분 1로 보일 수 있다. 즉 로직은 바뀌었지만 데이터가 희소하면 화면은 비슷하게 느껴질 수 있다. 다음에는 차트에 `계정 수`와 `총 태그 횟수`를 같이 보여주는 방식도 검토할 필요가 있다.

교훈

겉으로는 UI 문제처럼 보여도 실제로는 데이터 grain, 즉 데이터를 묶고 세는 단위의 문제일 수 있다. 다음부터는 화면이 이상해 보일 때 먼저 집계 기준과 쿼리 정의부터 확인해야겠다.


[문제 2. 비슷해 보이는 설정값들이 실제로는 서로 다른 역할을 하는 문제]

문제 포착

Snowflake 연결 문제와 dbt schema 문제를 다루는 과정에서 `.env`, Airflow Connection, Snowflake schema 설정이 모두 비슷한 "설정값"처럼 느껴졌다. 하지만 실제로는 각각 다른 위치에서 다른 목적으로 쓰이고 있었다.

문제 상황 파악

이걸 명확히 구분하지 않으면 원인을 잘못 짚을 가능성이 크다. 예를 들어 Airflow 적재가 실패했는데 `.env`를 먼저 의심하면 시간을 낭비할 수 있다. 반대로 dbt 결과가 이상한데 Airflow Connection만 확인하면 문제를 못 찾는다. 실제 운영에서는 이런 혼선 때문에 장애 대응 시간이 길어지는 경우가 많다.

해결 방법 도출

- 현재 구조를 그대로 두고 필요할 때만 직접 확인하는 방법
- 문서만 보강하는 방법
- 코드, 문서, 운영 규칙에서 역할을 분명히 나누는 방법

해결 방법 선정 및 타당한 이유

세 번째 방법을 선택했다. 문서만 추가하는 것은 빠르지만, 실제 코드 흐름과 연결되지 않으면 나중에 다시 헷갈릴 수 있다. 반대로 구조를 전부 새로 바꾸는 것은 지금 단계에서는 비용이 크고 위험하다.

그래서 이번에는 "Airflow 적재는 `snowflake_conn`", "dbt와 Streamlit은 `.env`", "Snowflake의 STAGE/MART는 저장 위치"처럼 책임을 분리해서 정리했다. 이 방법은 기존 구조를 크게 흔들지 않으면서도 원인 추적 속도를 높일 수 있어 운영 비용 대비 효과가 가장 크다고 판단했다.

예시 코드

def get_snowflake_connection():
    return snowflake.connector.connect(
        account=os.getenv("SNOWFLAKE_ACCOUNT"),
        user=os.getenv("SNOWFLAKE_USER"),
        password=os.getenv("SNOWFLAKE_PASSWORD"),
        warehouse=os.getenv("SNOWFLAKE_WAREHOUSE"),
        database=os.getenv("SNOWFLAKE_DATABASE"),
        schema=os.getenv("SNOWFLAKE_SCHEMA"),
        role=os.getenv("SNOWFLAKE_ROLE"),
    )

이 코드는 Streamlit과 dbt 쪽이 `.env`를 기준으로 연결된다는 점을 보여준다. 반면 Airflow 적재는 UI의 Connection 설정을 기준으로 움직인다. 이 차이를 회고 안에 명시적으로 남기는 것이 중요하다고 느꼈다.

성과 측정

- 변경 전:
  - 어떤 문제가 `.env` 때문인지, 어떤 문제가 Airflow Connection 때문인지 혼동 가능성이 컸음
- 변경 후:
  - `snowflake_conn` 기준 통일
  - `FSH`, `RAW_DATA`, `STAGE`, `MART` 역할 문서화
  - 실행 환경별 기준값을 설명으로 분리

검증 방식과 남은 리스크

검증은 실제 Airflow와 Streamlit이 서로 다른 설정을 참조하는지 로그와 연결 결과로 확인하는 방식으로 진행했다. Airflow는 Connection 이름을 기준으로 동작했고, Streamlit은 `.env`에 따라 결과가 바뀌는 것을 확인했다.

남은 리스크는 문서와 실제 설정이 다시 어긋날 수 있다는 점이다. 다음에는 `.env.example`과 Connection 설정 체크리스트를 더 명시적으로 맞추는 작업이 필요하다.

교훈

실무에서는 설정 파일을 많이 안다고 잘하는 것이 아니라, 각 설정이 어느 실행 흐름에서 쓰이는지를 구분할 줄 알아야 한다. 다음에는 설정을 볼 때도 "이건 어디서 읽히는 값인가?"를 먼저 묻는 습관을 들여야겠다.


[문제 3. 실행 시간 문제처럼 보였지만 실제로는 날짜 해석 문제였던 부분]

문제 포착

Airflow DAG가 한국 시간 기준으로는 자정 이후에 도는데, 수집 날짜가 하루 전처럼 보이는 현상이 있었다. 처음에는 스케줄이 잘못된 줄 알았다.

문제 상황 파악

실제 원인은 스케줄 자체보다 `logical_date`를 UTC 기준으로 해석하고 있었다는 점이었다. 즉, 몇 시에 실행됐는가보다 시스템이 그 날짜를 어떤 시간대로 읽고 있었는지가 더 중요한 문제였다. 이를 그대로 두면 수집 기준일이 하루씩 밀린 상태로 기록될 가능성이 있었다.

해결 방법 도출

- 스케줄 시간을 다시 조정하는 방법
- 로그 설명만 보강하는 방법
- `logical_date`를 KST로 변환해서 실제 처리 날짜를 계산하는 방법

해결 방법 선정 및 타당한 이유

세 번째 방법을 선택했다. 스케줄만 바꾸는 방법은 겉보기 시간은 맞출 수 있지만, 내부 해석이 여전히 UTC라면 같은 문제가 반복된다. 로그 설명만 추가하는 것도 임시방편에 가깝다. 반면 KST 변환은 코드 기준을 바로잡는 방식이라 한 번 수정하면 같은 해석이 전체 흐름에 일관되게 적용된다.

예시 코드

logical_date_kst = context["logical_date"].in_timezone("Asia/Seoul")
date_to_process = logical_date_kst.strftime("%Y-%m-%d")

짧은 코드지만, 날짜가 하루 밀려 보이는 문제를 해결하는 핵심이었다.

성과 측정

- 변경 전:
  - 실행 시각은 KST 자정 이후인데 처리 날짜는 전날처럼 보일 수 있었음
- 변경 후:
  - `DATE_TO_PROCESS KST` 로그 추가
  - 수집 기준일과 운영자 해석 기준이 일치하도록 수정

검증 방식과 남은 리스크

검증은 DAG 로그에서 `LOGICAL_DATE raw`, `LOGICAL_DATE KST`, `DATE_TO_PROCESS KST`를 같이 출력해 비교하는 방식으로 진행했다. 이 세 값이 기대한 흐름으로 이어지는지 직접 확인했다.

남은 리스크는 인스타 원천 데이터의 날짜 자체가 다른 시간대 의미를 가질 수 있다는 점이다. 즉 DAG 기준일을 KST로 맞췄다고 해서 원천 플랫폼의 표시 시간까지 완전히 같은 의미라고 단정할 수는 없다.

교훈

시간 문제는 실행 시각만 보면 안 되고, 시스템이 날짜를 어떻게 해석하는지도 같이 봐야 한다. 다음부터는 스케줄 이슈가 보이면 시간대와 logical date 해석부터 먼저 확인해야겠다.


[문제 4. dbt 모델이 돌아가기만 하면 된다고 생각하면 생길 수 있는 운영 문제]

문제 포착

dbt 모델을 점검하면서 테스트가 없고, 컬럼 이름이 실제 의미와 어긋나 있고, STAGE와 MART가 실제로 분리되지 않은 부분들이 눈에 띄었다. 겉보기에는 실행만 되면 괜찮아 보였지만, 운영 측면에서는 불안 요소가 많았다.

문제 상황 파악

이 상태를 그대로 두면 raw 데이터 품질 이상이 조용히 mart까지 퍼질 수 있고, 컬럼 이름 때문에 잘못된 해석이 반복될 수 있으며, 중간 결과와 최종 결과가 같은 공간에 쌓여 관리 책임이 모호해질 수 있다. 지금은 작게 보이지만, 나중에 데이터가 늘어나면 비용이 훨씬 커질 가능성이 있는 문제였다.

해결 방법 도출

- 일단 쿼리만 맞추고 나중에 정리하는 방법
- 테스트만 먼저 추가하고 구조는 그대로 두는 방법
- 테스트, naming, schema 분리를 함께 손보는 방법

해결 방법 선정 및 타당한 이유

세 번째 방법을 선택했다. 테스트만 추가하면 데이터 품질은 일부 보강되지만, 이름이 틀리거나 schema가 섞여 있으면 해석과 운영에서 다시 혼란이 생긴다. 반대로 구조만 바꾸고 테스트를 안 넣으면 이상 데이터를 빨리 잡기 어렵다. 한 번에 세 가지를 같이 잡는 것이 초기 비용은 조금 더 들지만, 이후 운영 비용을 가장 많이 줄이는 방식이라고 판단했다.

예시 코드

columns:
  - name: post_id
    tests:
      - not_null
      - unique

이런 형태의 테스트는 단순해 보여도 "게시물 ID가 비어 있지 않은가", "중복 저장되지 않았는가"를 자동으로 검증해준다.

성과 측정

- 변경 전:
  - source 테스트 부재
  - 이름이 실제 의미와 어긋나는 컬럼 존재
  - STAGE와 MART가 논리적으로만 분리되어 있었음
- 변경 후:
  - source 테스트 추가
  - naming 정리
  - STAGE/MART를 실제 schema로 분리

검증 방식과 남은 리스크

검증은 `dbt run`, `dbt test`, Streamlit 조회 위치 확인으로 진행했다. STAGE와 MART가 실제 다른 schema에 생성되는지, source 테스트가 정상 실행되는지, Streamlit이 MART를 기준으로 읽는지 확인했다.

남은 리스크는 모델 수가 늘어날수록 테스트 범위가 더 필요해진다는 점이다. 지금은 기본 테스트 중심이라 향후에는 날짜 범위 이상치, 계정명 정제 품질, post-level uniqueness 같은 항목도 더 보강해야 한다.

교훈

데이터 모델은 "돌아가는 SQL"보다 "나중에도 설명 가능하고 관리 가능한 구조"가 더 중요하다. 다음부터는 dbt 작업을 할 때도 기능 구현과 운영성을 처음부터 같이 보려고 한다.


[문제 5. Instagram DOM 변경에 대응하는 방식이 즉흥 패치로 끝날 위험]

문제 포착

Instagram 상세 페이지가 예전처럼 열리지 않거나 popup 로드가 실패하는 문제가 반복되었다. 처음에는 셀렉터 하나만 바꾸면 해결될 것처럼 보였다.

문제 상황 파악

하지만 외부 서비스 구조는 한 번 바뀌고 끝나는 것이 아니기 때문에, 이번에만 맞는 셀렉터를 억지로 찾아 붙이면 다음 변경 때 또 같은 문제가 생길 가능성이 컸다. 문제를 방치하면 이후에도 같은 유형의 장애를 계속 수동으로 추적해야 할 가능성이 있었다.

해결 방법 도출

- 당장 동작하는 셀렉터를 하나 찾아서 땜질하는 방법
- popup 대신 무조건 detail page만 쓰는 방법
- popup-first, detail fallback, URL/스크린샷 로깅을 같이 두는 방법

해결 방법 선정 및 타당한 이유

세 번째 방법을 선택했다. 셀렉터 땜질은 가장 빠르지만 다음 변경에 다시 취약하다. detail page만 강제하는 방식은 단순하지만 기존 popup 흐름이 살아 있을 때의 장점을 버리게 된다. 반면 popup-first + fallback 구조는 현재 동작 가능성을 유지하면서도, 실패 시 디버그 증거를 남겨 다음 대응 비용을 줄일 수 있다.

예시 코드

if popup_is_open:
    parse_popup()
elif current_url_contains_post:
    parse_detail_page()
else:
    save_debug_screenshot()

이 구조는 "한 가지 방식만 믿지 않는다"는 점에서 유지보수에 유리하다.

성과 측정

- 변경 전:
  - popup 실패 시 원인 파악이 어려웠음
- 변경 후:
  - popup-first + detail fallback 흐름 정리
  - 실패 시 URL, 스크린샷 로그 남김
  - v11 extractor 구조로 전환

검증 방식과 남은 리스크

검증은 실제 수집 로그에서 상세 진입 성공 여부, 디버그 스크린샷 생성 여부, URL 이동 흐름을 확인하는 방식으로 진행했다. 이전보다 실패 지점을 더 빨리 특정할 수 있게 된 것은 확인했다.

남은 리스크는 Instagram 구조가 다시 바뀔 수 있다는 점 그 자체다. 즉 이번 수정은 "영구 해결"이라기보다 "다음 문제를 더 빨리 찾을 수 있는 구조"에 가깝다.

교훈

외부 서비스에 의존하는 크롤러는 "한 번 고쳤다"보다 "다시 깨졌을 때 빨리 찾을 수 있다"가 더 중요하다. 다음부터는 셀렉터를 바꾸는 것만으로 끝내지 않고, 관측 가능한 구조를 같이 남겨야겠다.


[문제 6. 집계 결과만 보여주고 원본 근거를 같이 보여주지 않았던 점]

문제 포착

대시보드에서 숫자나 요약 문자열은 보였지만, 그 결과가 실제 어떤 게시물에서 나왔는지까지는 바로 확인하기 어려웠다. 사용자가 "왜 이 숫자가 나왔는지" 궁금해할 수 있는 상태였다.

문제 상황 파악

이 상태를 그대로 두면 대시보드는 결과만 보여주는 화면이 되고, 숫자의 신뢰를 설명하기 어려워진다. 특히 분석 도구는 단순히 예쁜 결과보다 "근거를 다시 확인할 수 있는가"가 중요하다. 원본 게시물 기준의 태그 계정 목록까지 내려가 볼 수 없다면 해석에 대한 불신이 생길 수 있었다.

해결 방법 도출

- 기존 요약 문자열만 유지하는 방법
- 계정별 전체 태그 목록만 보여주는 방법
- 게시물 단위까지 내려가서 게시물 1, 게시물 2처럼 태그 목록을 보여주는 방법

해결 방법 선정 및 타당한 이유

세 번째 방법을 선택했다. 요약 문자열만 남기면 화면은 단순하지만 설명력이 떨어진다. 계정별 전체 목록만 보여주는 방법도 중간 단계로는 괜찮지만, 실제로는 어느 게시물에서 나온 값인지까지 보이지 않는다. 게시물 단위로 내려가는 방식이 구현 비용은 조금 더 들지만, 결과를 신뢰할 수 있는 구조를 만드는 데 가장 직접적이라고 판단했다.

예시 코드

SELECT
    insta_id,
    post_id,
    post_date,
    tagged_insta_id
FROM fsh.raw_data.instagram_posts
WHERE insta_id = 'example_account'
ORDER BY post_date DESC;

이 결과를 화면에서
- 게시물 1 - 태그된 계정들
- 게시물 2 - 태그된 계정들
형태로 보여주면, 숫자만 보는 것이 아니라 근거까지 같이 확인할 수 있다.

성과 측정

- 변경 전:
  - 메인 표에서 숫자는 보이지만 원본 근거는 바로 확인하기 어려웠음
- 변경 후:
  - 메인 표는 핵심 열만 남김
  - `상세 보기` 선택 후 계정별 상세 패널 제공
  - 게시물 단위 태그 계정 목록까지 확인 가능

검증 방식과 남은 리스크

검증은 실제 계정을 선택했을 때 게시물별 태그 계정 목록이 raw 데이터와 같은지 보는 방식으로 진행했다. 적어도 계정별 근거를 다시 내려가 보는 흐름은 확보되었다.

남은 리스크는 태그 목록 문자열이 여전히 길 수 있다는 점이다. 앞으로는 게시물 카드형 UI나 더 읽기 쉬운 리스트 구조로 다듬을 필요가 있다.

교훈

데이터 대시보드는 숫자를 보여주는 것만으로 충분하지 않다. 그 숫자가 어디서 나왔는지 다시 내려가 볼 수 있어야 신뢰를 얻는다. 다음에는 결과 화면을 설계할 때부터 원본 근거 확인 경로를 같이 넣는 쪽으로 생각해야겠다.
